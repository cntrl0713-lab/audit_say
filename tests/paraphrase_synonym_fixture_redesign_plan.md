# 계획서: verify-paraphrase.ts 동의어 치환 시나리오 재설계

> 배경: `env 로딩 정비` 리뷰 중 처음으로 끝까지 실행된 `verify-paraphrase.ts`에서 시나리오 2(동의어 치환)의 두 문항이 6점(문턱 7점 미달)으로 실패 발견. "다음 종합리포트 5번 후보"로 제안했던 항목.
> 조사 완료: 두 실패 사례가 채점 엔진 결함이 아니라 테스트 픽스처 자체의 동의어 선정 오류임을 웹 검색으로 확인.

## 1. Goal

`tests/verify-paraphrase.ts` 시나리오 2는 "정답을 동의어로 치환해도 고득점(≥7점)이 보존되는지" 검증하려 했으나, 실제로 치환에 쓰인 단어들을 감사기준서 500(감사증거)·230(감사문서)의 실제 용어와 대조한 결과 **진짜 동의어가 아니었다**:
- Q4: "신뢰성"(감사증거의 질적 요소, 감사기준서 500의 표준 용어) → "신용성"(신용도·credit-worthiness — 재무/여신 분야 용어이며 감사기준서 어디에도 등장하지 않음)으로 치환. 이건 동의어가 아니라 **다른 개념의 단어**다.
- Q5: "감사인"(감사조서 소유권이 귀속되는 주체로 감사기준서 230이 명시하는 정확한 용어) → "감사수행팀"(실무를 수행하는 팀이라는, 소유권 귀속 주체와는 결이 다른 개념)으로 치환. "회계법인" → "감사회사"도 공식 법정 용어(공인회계사법상 회계법인)를 비공식 통칭으로 바꾼 것.
- Gemini의 채점 피드백("전문 용어 대신 일반 용어 사용", "잘못된 귀속 주체 서술")은 이 프로젝트의 채점 시스템 지시문(`lib/serverUtils.ts`)이 이미 명시한 1순위 규칙 — **"전문 용어의 정확성(필수): 의미가 비슷하더라도 일반적인 서술어는 인정하지 마십시오"** — 를 정확히 따른 것이다.

즉 이건 "채점이 부당하게 엄격한 버그"가 아니라 "테스트가 애초에 틀린 것을 동의어라고 착각해 넣은 것"이다. 이 계획은 시스템의 엄격한 전문용어 정책은 그대로 둔 채, 시나리오 2가 실제로 측정하려던 것(핵심 전문용어는 보존하면서 부차적 서술만 달라진 답안의 고득점 보존)을 제대로 측정하도록 픽스처를 다시 짠다.

## 2. Target behavior

변경 후 관찰 가능해야 하는 것:

- 시나리오 2(동의어 치환)의 새 답안 텍스트는 각 문항의 `keywords` 배열에 있는 핵심 전문 용어(예: Q4의 "관련성", "신뢰성", "충분성", "적합성" / Q5의 "최소 8년", "소유권", "감사인" 또는 "회계법인")를 원문 표기 그대로 포함하면서, 그 외 연결어·문장 구조만 달리 쓴다.
- 재실행 시 두 문항 모두 ≥7점을 받아, 이 시나리오가 원래 의도("핵심 개념 보존 시 정당한 고득점")를 실제로 검증한다.
- 채점 시스템(`lib/serverUtils.ts`의 systemInstruction, 필터, `computeRubricCoverage`)은 한 글자도 바뀌지 않는다 — "전문 용어의 정확성 필수" 정책은 이 프로젝트가 의도적으로 설계한 것이며, 이번 조사로 오히려 정상 작동이 재확인됐다.

변경되지 않아야 하는 것 (non-goals):

- **채점 프롬프트·필터·rubric 커버리지 로직**: 무변경. 이번 결함의 원인이 테스트 데이터였음이 확인됐으므로, 시스템 쪽을 느슨하게 바꾸는 건 32개 이상 v2 문항 전체의 전문용어 감별력을 떨어뜨리는 훨씬 큰 회귀 위험을 만든다.
- **시나리오 1(패러프레이즈: 어순·조사 변형)과 시나리오 3(반복 일관성)**: 이미 정상 통과(10점, 10점 / 편차 1점) — 무변경.
- **`verify-adversarial.ts`, `verify-shuffled.ts`**: 이번 계획과 무관.
- **v1/v2 실제 문제 데이터(`cpa_questions`, `cpa_questions_v2`)**: `verify-paraphrase.ts`의 픽스처는 이 파일 안에 하드코딩된 테스트 전용 데이터이며 실제 DB 문항과 무관하다 — 이번 수정은 DB에 전혀 영향을 주지 않는다.

## 3. Atomic requirements

- R1. `PARAPHRASE_FIXTURES`의 Q4(id 4) `synonymAnswer`를 재작성한다: 원문 keywords(`충분성`, `적합성`, `수량적 측면`, `질적 측면`, `관련성`, `신뢰성`)를 전부 정확한 표기로 포함하되, 그 외 서술(조사·연결어·문장 순서)만 원래 `modelAnswer`와 다르게 구성한다.
  - 예시 방향: `"감사증거의 충분성은 수량적 측면을, 적합성은 질적 측면 즉 관련성과 신뢰성을 나타내는 개념이다."` (최종 표현은 Slice 1에서 확정)
- R2. Q5(id 5) `synonymAnswer`를 재작성한다: `최소 8년`, `소유권`, `감사인`(또는 원문이 이미 병기하는 `회계법인`) 표기를 그대로 유지하고, "보존"→"보관" 등 keywords 배열에 없는 비핵심 서술어만 바꾼다.
  - 예시 방향: `"감사보고서일로부터 최소 8년 동안 보관해야 하며, 소유권은 감사인(회계법인)에게 속한다."`
- R3. 재작성된 두 답안 모두 `calculateMatchedCount` 사전 필터를 통과함을 확인한다(문턱: 5개 키워드 중 2개 이상 — 현재도 이미 통과하고 있었으므로 회귀 없음 확인 성격).
- R4. 재작성 후 `verify-paraphrase.ts`를 실행해 시나리오 2 두 문항 모두 ≥7점임을 실측 확인한다.
- R5. 이번 조사로 확인된 사실(신뢰성≠신용성, 감사인≠감사수행팀)을 `tests/verify-paraphrase.ts` 상단 주석 또는 픽스처 옆에 한 줄로 남겨, 다음에 이 파일을 다시 손대는 사람이 같은 착오로 "동의어"를 잘못 고르지 않도록 한다.

## 4. Open questions and assumptions

Blocking 없음.

- A1 (비차단): Q4·Q5의 정확한 대체 문장은 위 R1·R2에 방향만 제시했다 — 최종 문구는 Slice 1에서 실제로 다양한 표현을 시도해 확정한다(순수 텍스트 재작성이라 반복 조정 비용이 낮음).
- A2 (비차단): "핵심 전문용어를 전부 보존하면서 그 외만 바꾼다"는 시나리오 1(패러프레이즈: 어순·조사 변형)과 성격이 겹칠 수 있다 — 다만 시나리오 1은 keywords 표기 자체도 살짝 바뀌는(`관련성`→`~이 지니는 의의`처럼) 경우가 있는 반면, 재설계된 시나리오 2는 keywords 표기를 문자 그대로 보존하면서 그 주변 서술만 재구성한다는 점에서 여전히 구분되는 검증 축(연결어·전체 문장 구조 변형 vs 개별 어휘의 조사·형태 변형)으로 유지 가능하다고 가정한다.

## 5. Domain risks and edge cases

- **가장 큰 리스크는 "채점이 잘못됐다"고 오판해 시스템 프롬프트를 느슨하게 만드는 것** — 이번 조사로 그게 아님이 확인됐으므로, 이 계획은 의도적으로 `lib/serverUtils.ts`를 건드리지 않는다. 4절 비목표에서 이미 명시.
- **키워드 리터럴 보존과 "동의어스러움" 사이의 긴장**: R1·R2 방향대로 keywords를 그대로 유지하면 사실상 "핵심 용어는 그대로 두고 주변만 바꾸는" 시험이 되는데, 이는 이미 시나리오 1이 하는 것과 결이 비슷해질 수 있다(A2 참고). 완전히 다른 종류의 검증을 원한다면 이 시나리오 자체를 폐기하거나 목적을 재정의하는 것도 옵션이나, 이번 계획은 "최소 수정으로 기존 시나리오의 원래 의도를 살린다"는 범위로 좁혔다.
- **필터 통과 여부**: 재작성 시 keywords 표기를 원문 그대로 포함하므로 필터 통과는 오히려 더 안전해진다(원래 답안도 이미 통과했었음).

## 6. Affected boundaries

- **tests**: `tests/verify-paraphrase.ts`의 `PARAPHRASE_FIXTURES` 배열 중 Q4·Q5의 `synonymAnswer` 필드 2개 문자열, 주석 1줄 추가. 이것이 유일한 변경.
- domain logic / persistence / UI / API: 무변경.

## 7. Proposed implementation structure

변경 파일:

- `tests/verify-paraphrase.ts` — `PARAPHRASE_FIXTURES[0].synonymAnswer`(Q4), `PARAPHRASE_FIXTURES[1].synonymAnswer`(Q5), 상단 근거 주석. 이유: 결함의 유일한 원인 위치.

1차 패스에서 건드리지 말 것:

- `lib/serverUtils.ts`, `lib/rubric.ts` 등 채점 로직 전체.
- `PARAPHRASE_FIXTURES`의 다른 필드(`modelAnswer`, `keywords`, `paraphrased`, `question` 등) — synonymAnswer 2개만 수정.
- `verify-adversarial.ts`, `verify-shuffled.ts`.

## 8. Implementation slices

- **Slice 1 — 동의어 픽스처 재작성**
  - Goal: R1·R2·R5 적용.
  - Expected file scope: `tests/verify-paraphrase.ts` — 1파일, 2개 문자열 + 주석 1줄.
  - Why this slice is isolated: 순수 테스트 데이터 문자열 편집이며 다른 로직에 의존하지 않는다.
  - Coupled updates required: 없음.
  - Verification: `node -e`로 재작성된 두 문장에 대해 각 문항의 keywords 5개 전부가 리터럴로 포함되는지 문자열 포함 검사(정규화 후) — R3 사전 확인.
  - Done when: 두 문장 모두 keywords 전항목 포함 확인.

- **Slice 2 — 실측 검증**
  - Goal: R4 — 재작성된 답안이 실제로 ≥7점을 받는지 확인.
  - Expected file scope: 코드 변경 없음 (실행만).
  - Why this slice is isolated: Slice 1의 정적 확인이 끝난 뒤에만 실행 — 확실치 않은 문구로 API를 소모하지 않기 위해 분리.
  - Coupled updates required: 없음.
  - Verification (~2콜, 시나리오 2만 격리 실행하거나 전체 스크립트 실행 시 ~7콜):
    1. `npx tsx tests/verify-paraphrase.ts` 전체 실행 → 시나리오 2 두 문항 모두 "✅ 동의어 보존 성공"(≥7점).
    2. 시나리오 1·3도 회귀 없이 그대로 통과하는지 같은 실행 결과에서 재확인.
  - Done when: 시나리오 2가 2/2 PASS, 시나리오 1·3은 기존과 동일한 결과 유지. 만약 재작성 후에도 7점 미만이 나오면 문구를 더 조정해 재시도(같은 슬라이스 내 반복) — 그래도 계속 낮으면 Slice 1로 돌아가 다른 표현을 시도하고, R2 문서화(A1)에 시행착오 내역을 남긴다.

## 9. Acceptance checklist

- [ ] Q4·Q5 `synonymAnswer`가 각자의 keywords 5개 항목을 문자 그대로 포함
- [ ] `verify-paraphrase.ts` 시나리오 2 두 문항 모두 ≥7점
- [ ] 시나리오 1·3 회귀 없음(패러프레이즈 9~10점대, 반복 편차 ≤2점 유지)
- [ ] `lib/serverUtils.ts` 등 채점 로직 diff 없음
- [ ] 근거 주석 1줄 추가로 재발 방지 문서화

## 10. Deferred work

- 시나리오 2가 시나리오 1과 성격이 겹치는 문제(A2)를 근본적으로 재설계(예: 진짜 일반 어휘 동의어가 존재하는 별도 질문으로 픽스처 자체를 교체)하는 것 — 이번 계획은 최소 수정으로 원래 의도를 복원하는 데 그친다.
- 이 파일(`verify-adversarial.ts`/`verify-paraphrase.ts`/`verify-shuffled.ts`)의 픽스처(qid 1~5)를 v2 문항 기준으로 전면 교체할지 여부 — 별도 논의 필요, 이번 범위 밖.
- "전문 용어의 정확성 필수" 정책 자체가 실제 KICPA 채점 기준과 얼마나 부합하는지에 대한 더 넓은 정책 검토 — 이번 계획은 정책을 확인만 하고 변경하지 않는다.
