# 문제 134 파일럿 종합 테스트 계획 (plan-strict)

> `cpa_questions_v2`에 적재된 문제 134(낮은 보수 및 제2의견 관련 윤리적 위협과 안전장치)를 파일럿으로 v2 데이터·필터·채점 전 계층을 관통 검증하는 계획. 134는 다물음(2개 sub, 배점 4+6), all 모드 물음 2개, 소수점 항목 배점(1.33), 파편 답안 원소("안전장치")를 모두 갖고 있어 v2 파이프라인의 시금석으로 적합하다. 여기서 확정된 테스트 패턴은 이후 나머지 문항 전수 검증의 템플릿이 된다.

---

## 0. 사전 검토에서 발견된 실제 위험 (테스트가 겨냥할 과녁)

계획 수립 중 `cpa_problems_v2.json`의 134 실데이터를 검토하며 확인한 사항:

- **W1. 범용어 variants 다수**: `"배정"`, `"입증"`, `"요청"`, `"설명"`, `"한계"`, `"위협"`, `"지침"`, `"준수"` — 2글자라 `validateRubric`의 최소 길이 검증(≥2자)은 통과하지만, 감사 답안 어디에나 나올 수 있는 범용어다. 특히 물음 1만 쓴 답안에도 "입증"이 있고, "요청"·"설명"은 물음 2의 항목(2-2, 2-3) variants이므로 **물음 간 교차 오염**과 **키워드 샐러드 방어 약화** 위험이 실재한다.
- **W2. 소수점 배점**: sub 2의 항목이 1.33/1.33/1.34로 분할됨. 합계 검증(1e-9 오차)은 통과하나, 부분(½) 인정 시 0.665점 같은 값이 생겨 반올림 정책과 충돌할 수 있다.
- **W3. "3가지 서술" vs `mode: "all"`**: 문제 문구는 "안전장치를 3가지 서술하시오"인데 sub 2는 윤리강령(2-1) + 안전장치 3개(2-2~2-4)를 `all`로 묶었다. 표준 답안의 안전장치가 정확히 3개뿐이라 결과적으로 등가지만, 의도된 설계인지 기록·확인이 필요하다.

## 1. Goal

문제 134 하나에 대해 ① 데이터 계층(스키마·루브릭·자기일관성) ② 필터 계층(variants 사전 필터) ③ 채점 계층(현행 홀리스틱 엔진 E2E) ④ 루브릭 판정 계층(엔진 구현 후)을 관통하는 테스트를 구축·실행하고, W1~W3의 실제 영향을 정량화한다. 데이터 결함이 발견되면 수정은 별도 결정(사람 검수)으로 넘긴다.

## 2. Target behavior

**완료 후 확보되는 것:**

- 134의 데이터 품질(자기 커버리지, 교차 오염도, 신구 변형도)이 수치로 기록된 상태
- 현행 채점 엔진에서 134가 정답·부분답·오답·적대 답안에 어떤 점수를 주는지 실측 베이스라인
- 나머지 문항에 그대로 재사용할 수 있는 검증 스크립트 패턴

**변경하지 않는 것 (Non-goals):**

- 채점 엔진 코드(`lib/serverUtils.ts`) — 여전히 v1 경로를 읽으며, 루브릭 판정 엔진 구현은 별도 계획(`rubric_grading_plan.md` Slice 3)
- 134의 데이터 자체 — 테스트는 결함을 드러낼 뿐, 루브릭/variants 수정은 결과 확인 후 사람이 결정
- `cpa_questions_v2` 테이블 스키마

## 3. Atomic requirements

- R1. 134 행이 `validateCpaQuestionV2` + `validateRubric`을 통과함을 테스트로 고정 (소수점 배점 합산 포함)
- R2. 물음 정합: 문제 설명의 번호 개수(2개) = rubric sub 개수, sub 배점 합 4+6=10
- R3. 자기 커버리지: 각 item의 variants 중 최소 1개가 모범답안 전문(배열 join, 공백 제거 정규화)에 매칭됨
- R4. 교차 오염 정량화: 물음 1의 모범답안 텍스트만으로 물음 2의 각 item variants가 몇 개 매칭되는지(및 역방향) 리포트 — W1의 실측
- R5. 범용어 variants 리포트: 전체 v2 문항(현재 10개)의 variants 중 여러 문항에 반복 등장하는 단어 상위 목록 — W1의 전수 버전
- R6. 신구 대조: v1 문제 134과 v2의 question_description/model_answer 텍스트 bigram Jaccard 유사도 기록 (과도 변형 여부 확인 — 134는 원문 보존이 원칙이었으므로 높아야 정상)
- R7. 필터 시뮬레이션: variants 평탄화를 키워드로 쓰는 사전 필터에 대해 ① 모범답안 전문 통과 ② 물음 1만 통과 ③ 물음 2만 통과 ④ 무관 텍스트(다른 chapter 주제) 차단 여부 — ④가 범용어 탓에 뚫리면 발견사항으로 기록
- R8. 현행 엔진 E2E: BatchItem을 v2 콘텐츠로 직접 구성(q=v2 설명, m=v2 모범답안 join, k=variants 평탄화)해 `gradeBatch` 호출, 아래 시나리오 매트릭스 실측
- R9. 반복 일관성: 모범답안 전문 3회 채점 편차 ≤2
- R10. 루브릭 판정 엔진 완성 후의 정밀 기대값 정의 (지금은 문서로 고정만, 실행은 이연)

## 4. Open questions and assumptions

**사람 결정 필요 (테스트 결과 확인 후):**

- Q1. sub 2 배점을 정수화할 것인가 — 예: 윤리강령 3점 + 안전장치 3×1점 = 6점으로 재배분하면 소수점(W2)이 사라짐. 루브릭 판정 엔진의 반올림 정책 설계 전에 결정 권장
- Q2. 범용어 variants(W1)를 구 단위로 보수할 것인가 — 예: `"입증"` → `"배정됨을 입증"`, `"요청"` → `"논의 허용 요청"` (테스트 R4/R7 결과가 판단 근거)
- Q3. 정답 내용에 프롬프트 주입 문구가 섞인 답안을 전체 0점 처리하는 현재 정책(`lib/serverUtils.ts:171`)을 그대로 유지할 것인가, 아니면 주입 시도만 무시하고 실제 답안 내용만큼 채점하도록 완화할 것인가 — S8 실측(정답+주입 → 0점)을 근거로 `verify-134-e2e.ts`의 기대값을 0~0으로 고정해뒀으나, 이는 "현재 코드 동작을 기준선으로 잠정 확정"한 것일 뿐 정책 자체가 검토·합의된 것은 아님. 루브릭 판정 엔진 설계 전에 결정 필요

**안전한 가정:**

- A1. E2E는 현행 홀리스틱 엔진(`gemini-3.1-flash-lite`) 기준 베이스라인이다 — 물음별 배점을 엔진이 모르므로 부분답 점수는 "정확값"이 아닌 "허용 범위"로 판정
- A2. API 비용: 시나리오 9종 × 1회 + 반복 3회 ≈ 12콜 이내
- A3. v2 로컬 파일(`cpa_problems_v2.json`)과 DB 행이 일치한다고 가정하되, 테스트 스크립트는 DB에서 직접 읽어 이 가정 자체도 검증(불일치 시 리포트)

## 5. Domain risks and edge cases

- **"기존공인회계사" 동의어**: 수험생이 "전임 감사인", "기존 감사인"으로 쓸 수 있음 — variants에 없으므로 필터·판정에서 손해 가능. E2E 시나리오에 동의어 변형 정답 포함하되, 어떤 동의어가 실무상 통용되는지는 needs accounting verification 표기
- **물음 2의 복합 구조**: 윤리강령 식별(지식)과 안전장치 나열(열거)이 한 물음 — 윤리강령만 빼먹은 답안(안전장치 3개 완벽)이 4/6을 받는 게 도메인 관점에서 적정한지 검수 포인트
- **파편 원소 "안전장치"**: 모범답안 5번째 원소가 제목 조각 — join 시 자기 커버리지(R3)에는 무해하나, 모범답안을 화면에 보여줄 때의 품질 문제로 별도 기록
- **소수점 점수 저장**: 1.33 기반 산술이 나중에 `saveQuizNoteAction`(score number)에 흘러갈 때 표시 정밀도 — 루브릭 엔진 설계 시 반올림 정책과 함께 결정

## 7. Proposed implementation structure

- `tests/verify-v2-quality.ts` — 문항 id를 인자로 받는 범용 구조(기본 134). 추후 전수 실행 시 인자만 바꿔 재사용
- `tests/verify-134-e2e.ts` — 시나리오 픽스처는 파일 상단에 상수로 정의해 사람이 검토·수정 가능하게

**변경 금지**: `lib/*` 전체, `cpa_problems_v2.json`, DB 데이터

## 8. Implementation slices

- **Slice 1 — 데이터 계층 검증 스크립트 (무비용)**
  - Goal: R1~R6. W1(교차 오염)·W2(소수점 통과 여부)·W3(구조 기록)을 수치로 확정
  - Expected file scope: `tests/verify-v2-quality.ts` (신규 1개)
  - Why this slice is isolated: DB 읽기 전용 + 순수 계산, API 불필요
  - Coupled updates required: 없음
  - Verification: `npx tsx tests/verify-v2-quality.ts 134` 실행 → 리포트에 ① 검증 통과 여부 ② 자기 커버리지 item별 매칭 variants ③ 교차 오염 매트릭스(sub1 텍스트 × sub2 variants) ④ 신구 Jaccard ⑤ 전 문항 범용어 상위 목록 출력
  - Done when: W1이 "물음 1 답안이 물음 2 항목 N개에 매칭" 형태의 수치로 확정되고, R3에서 커버리지 0인 item이 있으면 목록화됨

- **Slice 2 — 필터 계층 시뮬레이션 (무비용)**
  - Goal: R7. variants 기반 사전 필터가 정답을 살리고 무관 답안을 막는지 로컬 확인
  - Expected file scope: `tests/verify-v2-quality.ts`에 섹션 추가 (같은 파일)
  - Why this slice is isolated: `calculateMatchedCount` 재사용, 결정적
  - Coupled updates required: 없음
  - Verification: 4개 입력(전문/물음1만/물음2만/무관 텍스트) 각각의 매칭 수·통과 여부 표 출력
  - Done when: 무관 텍스트가 범용어 탓에 통과하는지 여부가 확정됨 (통과하면 Q2 판단 근거로 기록)

- **Slice 3 — 현행 엔진 E2E (API, ~12콜)**
  - Goal: R8~R9. 아래 매트릭스 실측
  - Expected file scope: `tests/verify-134-e2e.ts` (신규 1개)
  - Why this slice is isolated: gradeBatch 직접 호출, 프로덕션 코드 무접촉
  - Coupled updates required: 없음
  - Verification — 시나리오 매트릭스:

    | # | 답안 | 기대 |
    | --- | --- | --- |
    | S1 | 모범답안 전문 | ≥9 |
    | S2 | 물음 1만 (안전장치 2개) | 3~5 (배점 4 근사) |
    | S3 | 물음 2만 | 5~7 (배점 6 근사) |
    | S4 | 물음 2에서 윤리강령 누락, 안전장치 3개만 | 3~5 |
    | S5 | 물음 2 안전장치 2개만 (3가지 요구) | S3보다 명확히 낮음 |
    | S6 | variants 전체 나열 (키워드 샐러드) | ≤3 |
    | S7 | 반대 결론 ("낮은 보수 수임엔 안전장치 불필요, 제2의견은 제한 없이 제공 가능") | 0 |

    | S9 | S1 3회 반복 | 편차 ≤2 |

  - Done when: 9개 시나리오 점수와 피드백이 기록되고, 기대 범위 이탈 건은 원인 메모(엔진 한계 vs 데이터 결함)와 함께 분류됨

- **Slice 4 — 루브릭 판정 엔진용 정밀 기대값 고정 (문서만, 실행 이연)**
  - Goal: R10. 엔진 구현 후 같은 픽스처로 실행할 정밀 기대값을 지금 확정
  - Expected file scope: `tests/verify-134-e2e.ts` 상단 주석 또는 본 문서 갱신
  - Why this slice is isolated: 코드 실행 없음, 기대값 정의만
  - Coupled updates required: 없음
  - Verification — 정밀 기대값: S2=4.0 / S3=6.0 / S4=4.0 / S5≈4+α(1.33×2 반올림 정책에 따름 — Q1 결정 필요) / 물음 1 텍스트만으로 물음 2 항목 득점 0 (중복·범용어 인용 차단 검증)
  - Done when: Q1(배점 정수화) 결정과 함께 기대값이 확정 문서화됨

**실행 순서**: 1 → 2 → 3 (→ 4는 루브릭 엔진 구현 후). Slice 1~2는 API 키 불필요.
