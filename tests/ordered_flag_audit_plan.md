# 계획서: ordered 플래그 후속 — 31문항 순차성 감사 및 선별적 플래그 부여

> 배경: `tests/ordered_rubric_flag_plan.md` 10절 이연 항목. 인프라(스키마·주입·저작 규칙)는 구현·검증 완료, 현재 플래그 부여 문항은 307 하나뿐.
> 이 계획의 산출물은 코드가 아니라 **판정 근거가 남는 데이터 변경**이다 — 감사(audit) 문서 + 승인된 문항의 `ordered: true` 반영 + 실측.

## 1. Goal

v2 나머지 31문항의 모범답안을 전수 판독해 "순차 절차"(항목 간 선후관계가 답의 실질인 문항)를 선별하고, 사용자 승인을 거친 문항에만 `ordered: true`를 부여한 뒤 문항별 역순 프로브로 감점 작동을 실측한다. 감사 과정에서 다물음 문항 중 일부 sub만 순차인 사례가 나오면, 지금까지 단위 테스트로만 고정돼 있던 혼합형(`buildOrderedNotice`의 "물음 N" 분기) 실측도 함께 해소한다. 잘못 플래그를 달면 열거형 답안이 부당 감점되는 방향(학생 피해)이므로, 전 과정의 기본값은 **"애매하면 달지 않는다"**(현행 순서 불감이 관대한 쪽 오차)이다.

## 2. Target behavior

변경 후 관찰 가능해야 하는 것:

- 31문항 각각에 대해 순차성 판정(대상/비대상/애매→비대상)과 그 근거 한 줄이 감사 문서에 기록되어, 나중에 어떤 문항이 왜 플래그를 받았는지/안 받았는지 추적 가능하다.
- 사용자가 승인한 문항만 로컬 JSON·DB 양쪽에 `ordered: true`가 반영된다.
- 신규 플래그 문항 각각에 대해: 역순 답안 ≤8점(감점 작동), 정순 모범답안 9~10점(오발동 없음)이 실측된다.
- 플래그를 받지 않은 문항들의 채점 동작은 이전과 동일하다 (스팟 확인).
- (감사 결과에 혼합형 후보가 있는 경우) 일부-sub-ordered 문항의 실채점에서 "물음 N" 지시 분기가 실제로 작동함이 실측된다 — `ordered_rubric_flag_plan.md` A2 해소.

변경되지 않아야 하는 것 (non-goals):

- **코드 일체 무변경**: `lib/rubric.ts`·`lib/serverUtils.ts`·검증 로직·지시 문구는 이미 검증된 상태 그대로. 이 계획은 순수 데이터+문서+실측 라운드다. (실측용 프로브 스크립트 1개 신규 작성은 tests/ 안에서만.)
- **307**: 이미 반영·실측 완료 — 재작업 없음.
- **v2에 없는 문항(906 등)**: 신규 문항 저작은 Perplexity 워크플로우의 몫이며, 저작 규칙(3단계 ordered 지침)은 이미 반영돼 있으므로 이 계획의 범위 밖.
- **판정이 "애매"로 남은 문항**: 플래그를 달지 않고 감사 문서에 애매 사유만 기록 — 근거 없이 추정으로 달지 않는다.

## 3. Atomic requirements

- R1. 기계적 사전 스크리닝: 31문항의 모범답안·문제문에서 순차 신호(조건 연결어 "만약/~라면/그러한", 에스컬레이션 표지 "거부 시/미해결 시/~한 후/이후/다음으로", 절차 지시 "단계") 검출을 스크립트로 수행해 후보 목록을 뽑는다. 신호가 없는 문항도 전수 목록에는 포함한다(스크리닝은 정독 우선순위를 정할 뿐, 판정을 대신하지 않음).
- R2. 전수 정독 판정: 31문항 각각에 대해 모범답안을 실제로 읽고 아래 기준으로 3분류한다 — **대상**(항목 간 명시적 선후관계: 앞 항목의 결과·조건이 뒤 항목의 전제이거나, 기준서가 절차 수행 순서 자체를 규정), **비대상**(병렬 열거: 유형·사유·정의 나열), **애매**(근거 불충분 → 비대상 처리 + 사유 기록).
- R3. 감사 문서 산출: `tests/ordered_audit_results.md`에 문항별 [id, 판정, 근거 한 줄, (대상인 경우) ordered 부여할 sub 번호] 표를 기록한다.
- R4. **승인 게이트**: R3의 "대상" 목록을 사용자에게 제시하고 승인받은 문항만 다음 단계로 진행한다 (채점 정책에 직접 영향을 주는 데이터 변경이므로 자동 진행 금지).
- R5. 데이터 반영: 승인 문항의 해당 sub에 `ordered: true` 추가 → `validateCpaQuestionV2` 통과 확인 → `upload_cpa_v2.ts` dry-run → `--apply`.
- R6. 문항별 실측 프로브: 신규 플래그 문항 각각에 대해 역순 답안(플래그 sub의 항목들을 역순 재배열, 다물음이면 해당 sub만 역순) 채점 ≤8점 + 정순 S1(verify-v2-e2e LIGHT) 9~10점을 확인한다.
- R7. 혼합형 실측(조건부): 승인 문항 중 다물음-일부-sub-ordered가 있으면, 그 문항의 채점 요청에 "물음 N" 형태 지시가 주입됨을 확인하고(R6의 역순 프로브가 이를 겸함) 결과를 감사 문서에 기록한다. 해당 사례가 없으면 "혼합형 실측은 여전히 데이터 대기"로 명시 기록.
- R8. 무플래그 회귀 스팟: 플래그를 받지 않은 열거형 대표 2문항(예: 316, 122) LIGHT 재실행 [10,0,0,0] 유지 확인.
- R9. 완료 후 메모리(`grading-policy-rulings.md`)의 "31문항 감사 미완" 문구를 결과(부여 문항 목록)로 갱신한다.

## 4. Open questions and assumptions

Blocking:

- **Q1 — 승인 게이트(R4)는 계획 실행 중간에 사용자 입력이 필요하다.** Slice 2 완료 시점에 "대상" 목록과 근거를 제시하고 멈춘 뒤, 승인받은 목록으로만 Slice 3을 진행한다. 구현자는 이 게이트를 건너뛰고 전체 후보에 플래그를 달아서는 안 된다.

Blocking 아님 — 아래 가정으로 진행:

- A1: 판정 주체는 구현 패스(정독+근거 기록)이되 최종 결정은 사용자(R4)다. 구현 패스의 판정이 확신 없으면 "애매→비대상"이 기본값.
- A2: 순차성의 조작적 정의는 R2의 기준을 따른다. 참고 확정 사례 — 대상: 307("만약 그러한 위험이 있다면"으로 연결), 906류 물음1(요청→거부 시→미해결 시). 비대상: 316(유형 4가지), 122(예외 사유 열거), 313(구성요소 5가지), 200(상황 열거).
- A3: 역순 프로브의 기대치 "≤8"은 감점 여부만 본다(307 실측 3점이었지만 문항별 편차 허용). 정순 기대치는 verify-v2-e2e S1 기준(9~10) 그대로.
- A4: 실측 프로브는 기존 `verify-item-reorder.ts`의 조립 헬퍼를 일반화한 소형 스크립트(신규, CLI로 id 지정) 또는 문항별 1회성 실행으로 충분하다 — 코드 산출물의 형태는 구현 패스 재량이되 tests/ 밖은 금지.

## 5. Domain risks and edge cases

- **오분류의 비대칭 피해**: 열거형에 잘못 플래그를 달면 학생의 정당한 순서 자유가 감점된다(부당 감점 — 확정 정책 위반). 반대로 순차 문항을 놓치면 현행(관대) 유지일 뿐이다. 따라서 임계는 보수적으로: 명시적 언어 근거(조건 연결어, 기준서상 절차 순서) 없이 "상식적으로 순서가 있어 보인다" 수준이면 비대상.
- **부분 순차 sub**: 다물음 문항에서 한 sub만 순차인 경우 해당 sub에만 플래그 — 문항 전체에 달면 다른 열거형 sub까지 순서 감점 지시가 걸린다(혼합형 문구는 "물음 N"을 명시하므로 지시 자체는 안전하지만, 잘못된 sub 지정은 그대로 오발동으로 이어짐).
- **역순 프로브의 자기모순 함정**: 열거형으로 판정된 문항을 실수로 프로브 대상에 넣으면 "감점 안 됨"이 결함처럼 보인다 — 프로브는 반드시 승인 목록만 대상으로.
- **DB upsert 범위**: `upload_cpa_v2.ts`는 전 문항 순회 upsert이므로 승인 외 문항도 동일 내용 재적재된다(멱등, 기존 확인됨). dry-run에서 검증 실패 0건 선확인.
- **감사 문서의 증거성**: 근거 없는 판정 기록("순차임" 한 마디)은 다음 라운드에서 재검토 비용을 만든다 — 근거는 모범답안의 실제 문구 인용으로.

## 6. Affected boundaries

- **persistence**: `cpa_uploader/data/cpa_problems_v2.json`(승인 문항의 sub별 ordered 필드) + DB 재업로드. DDL 없음.
- **tests**: 신규 감사 문서 `tests/ordered_audit_results.md`, 실측 프로브 스크립트 1개(신규, tests/).
- domain logic / UI / API / 스키마: **무변경** (인프라는 전 라운드에서 완결).

## 7. Proposed implementation structure

변경/신규 파일:

- `tests/ordered_audit_results.md` (신규) — R1 스크리닝 결과 + R2 전수 판정표 + R6~R8 실측 기록. 이 계획의 중심 산출물.
- `cpa_uploader/data/cpa_problems_v2.json` — 승인 문항의 ordered 필드만 (R5).
- `tests/verify-ordered-probe.ts` (신규, 이름 재량) — 승인 문항 대상 역순 프로브 (R6). `verify-item-reorder.ts`의 조립 헬퍼 재사용 가능. 관례 준수: `loadEnvLocal()` + `gradeBatch` 동적 import + `r: JSON.stringify(rubric)`.

1차 패스에서 건드리지 말 것:

- `lib/` 전체 (rubric.ts의 문구·분기 포함 — 혼합형 실측에서 이탈이 나와도 이 계획 안에서 코드 수정 금지, 별도 라운드로)
- `tests/verify-item-reorder.ts`·`verify-order-rule-prototype.ts` (기존 실측 기록 보존)
- `rubric_extraction_prompt.md` (저작 규칙 이미 반영됨)
- 307 데이터

## 8. Implementation slices

- **Slice 1 — 기계적 스크리닝 (오프라인, API 0콜)**
  - Goal: R1 — 31문항 전수에 대해 순차 신호 검출 결과를 뽑아 정독 우선순위를 만든다.
  - Expected file scope: `tests/ordered_audit_results.md` 초안 (스크리닝 표까지만). 스크립트는 일회성 node -e 수준이면 파일로 남기지 않아도 됨.
  - Why this slice is isolated: 순수 텍스트 분석 — 어떤 동작도 바꾸지 않음.
  - Coupled updates required: 없음.
  - Verification: 표에 31개 행이 전부 존재하는지(전수성), 307이 제외돼 있는지.
  - Done when: 문항별 [신호 검출 여부, 검출 문구] 표 완성.

- **Slice 2 — 전수 정독 판정 + 승인 요청 (오프라인, API 0콜)**
  - Goal: R2·R3 — 31문항 모범답안을 실제로 읽고 3분류 판정표를 완성, "대상" 목록을 사용자에게 제시.
  - Expected file scope: `tests/ordered_audit_results.md` (판정표 완성).
  - Why this slice is isolated: 판단 작업이며, R4 승인 게이트 앞에서 반드시 멈춘다.
  - Coupled updates required: 없음.
  - Verification: 모든 "대상" 판정에 모범답안 실제 문구 인용 근거가 붙어 있는지, "애매"가 전부 비대상 처리됐는지.
  - Done when: **사용자에게 대상 목록·근거 제시 완료 (여기서 중단, 승인 대기)** — 이 슬라이스의 완료 조건은 구현이 아니라 보고다.

- **Slice 3 — 승인 문항 데이터 반영**
  - Goal: R5 — 승인된 문항만 ordered 부여 + DB 반영.
  - Expected file scope: `cpa_problems_v2.json` (승인 문항 행들만).
  - Why this slice is isolated: 승인 게이트 이후에만 실행. 데이터 diff가 승인 목록과 1:1인지 기계적으로 확인 가능.
  - Coupled updates required: 없음.
  - Verification: 승인 목록 외 문항에 ordered가 없는지 grep으로 확인 → `validateCpaQuestionV2` 전건 0오류 → dry-run 통과 → `--apply` → DB 재조회로 반영 확인.
  - Done when: 로컬·DB 양쪽에서 승인 목록과 플래그 상태가 정확히 일치.

- **Slice 4 — 실측 및 기록 (문항당 ~5콜)**
  - Goal: R6·R7·R8 — 신규 플래그 문항별 역순 ≤8 / 정순 9~10, 혼합형 분기 실측(해당 시), 무플래그 회귀 스팟. 이탈 발견 시 이 슬라이스에서 코드·데이터 수정 금지 — 원인 기록 후 별도 라운드.
  - Expected file scope: `tests/verify-ordered-probe.ts`(신규), `tests/ordered_audit_results.md`(실측 기록 추가).
  - Why this slice is isolated: 검증 전용.
  - Coupled updates required: 없음.
  - Verification: 문항별 [역순 점수, 정순 점수, 판정] 표 + 316·122 LIGHT [10,0,0,0] + `npm run typecheck`(신규 스크립트 포함) + `npm test` 무회귀.
  - Done when: 감사 문서에 전 실측 기록 완비, 이탈 0건(또는 이탈이 원인 분석과 함께 기록되고 후속 라운드로 이관).
